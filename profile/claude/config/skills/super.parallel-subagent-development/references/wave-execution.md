# Multi-task wave execution

Full per-step procedure for a `parallel` wave (`waveKinds[i] === "parallel"`). Run the tasks concurrently, each isolated — worktrees are mandatory here (per the Worktree invariant), not optional. Paths use the base directories shown in the skill context header.

1. **Set status** — for every task in the wave, set its `task-NN.md` to `**Status:** IN_PROGRESS` on disk before dispatching. Prefer the deterministic, idempotent script: `node <super.subagent-driven-development-base-dir>/scripts/mark-task-status.cjs --task-file <task-NN.md> --status IN_PROGRESS`. Fallback: edit the line manually.
2. **One worktree per task** — for each task, create an isolated worktree (see `./integrating-worktrees.md`), each branching off the base branch with its own setup + clean baseline.
3. **Dispatch implementers in parallel** — one implementer per task, each pinned to its own worktree path, each carrying the full task text + PRD + Spec (read the `**PRD:**`/`**Spec:**` fields and include their content; never make the subagent read the plan). **PRD/Spec guard (same as the sequential skill):** per task, `**Spec:**` must be present and resolve on disk — if missing, empty, or unresolvable, **stop and resolve it before dispatching that task**. `**PRD:**` is optional: `N/A` is valid (pass `N/A` to the implementer); only a non-`N/A` PRD path that fails to resolve is an error. Check this per task *before* the fan-out, since one task's missing context would otherwise ship blind across the whole wave. Use `super.subagent-driven-development/agents/implementer.md` as the base plus the worktree scoping block from `../agents/implementer.md`. The implementer implements, tests, commits **inside its worktree**, then self-reviews.
4. **Two-stage review per task** — after each implementer returns, run the spec compliance reviewer then the code quality reviewer against that task's worktree diff, using `super.subagent-driven-development/agents/spec-reviewer.md` and `super.subagent-driven-development/agents/code-quality-reviewer.md`. Review loops stay inside the worktree; these per-task reviews can run concurrently across the wave.
5. **Integrate** — once every task has passed both reviews, run the deterministic pre-merge gate before touching the base branch: `node <super.parallel-subagent-development-base-dir>/scripts/check-integration.cjs --base <base-branch> --feature <feature-name> --tasks <wave-task-numbers> --repo-root "$(git rev-parse --show-toplevel)"`. Merge **only if** it reports `readyToIntegrate: true`; treat each `blocker` as a hard stop and each `missedDependencySignal` as a conflict to inspect first (details in `./integrating-worktrees.md § Pre-merge gate`). Then integrate the worktrees one at a time, following the gate's `mergeOrder` and the conflict handling in `./integrating-worktrees.md`. Integration is sequential by design.
6. **Verify the merged result** — after all worktrees are integrated, run `super.verification-before-completion` against the **base branch** (full test suite), not just the per-worktree results. Independent tasks can interact once merged; this gate catches integration bugs a per-worktree run cannot.
7. **Update the tracker** — only after the merged result verifies, make the two physical edits per task on the base branch: `- [ ] N.` → `- [x] N.` in `tasks-<feature-name>.md`, and `Status: IN_PROGRESS` → `Status: DONE` in `task-NN.md`. Prefer the deterministic script, which does both atomically: `node <super.subagent-driven-development-base-dir>/scripts/mark-task-status.cjs --task-file <task-NN.md> --status DONE --tasks-index <tasks-<feature-name>.md> --task-number N`. This is the authoritative completion record `super.finishing-a-development-branch` reads.
8. **Clean up worktrees** — remove the wave's worktrees (or let the native tool reclaim them) before starting the next wave, so stale worktrees don't accumulate.

## BLOCKED / failed-review handling

If a task is BLOCKED or fails review repeatedly, use the same status logic as the sequential skill (provide context and re-dispatch, escalate model, split the task, or escalate to the human) — including the **same review-loop cap**: a reviewer that rejects across ~3 non-converging rounds is treated like a `BLOCKED` implementer and escalated, never looped forever. A verified bug or failing test surfacing during implementation is a debugging problem, not a review nit — route it to **super.systematic-debugging** (its capped 4-phase loop), as the sequential skill's § Handling Implementation Bugs, then resume the task's gates. Do not let one stuck task block its independent siblings — finish and integrate the siblings, then resolve the blocked one before declaring the wave done.
